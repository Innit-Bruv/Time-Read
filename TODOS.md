### ~~TODO-002: Fix broken Celery fallback in ingest.py~~ ✅ DONE
**What:** Restructure the synchronous fallback in `ingest.py:62-70` to call the pipeline logic directly rather than the Celery task wrapper.
**Why:** When Celery/Redis is unavailable, the current fallback calls `process_content_task(str(content.id))` synchronously. This works for the happy path but crashes when `self.retry()` is invoked inside the task (it requires a Celery runtime context). The outer `except Exception: pass` swallows the crash silently. Content is permanently stuck at `"pending"` with no log and no user feedback.
**Pros:** Working ingestion pipeline even without Redis running (dev mode, Railway startup).
**Cons:** Requires extracting pipeline logic into a standalone function callable without Celery context.
**Context:** The fix is to extract `fetch_and_extract → segment → embed → save` logic from `process_content_task` into a helper function (e.g., `run_pipeline(content_id, db)`) and call that directly in the fallback. The Celery task body calls the same helper via `.delay()`.
