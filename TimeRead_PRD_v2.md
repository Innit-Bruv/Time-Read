# PRD — TimeRead: Personal Time-Aware Reading System
### Version 2.0 — AI Coding Agent Optimized

---

## 1. Product Overview

**TimeRead** is a single-user web application that saves reading content from Substack, Twitter/X, and articles, then surfaces them as time-optimized reading packs.

**Primary user query:** "I have 15 minutes" → system returns a curated reading pack that fits exactly within that window.

**User environment:**
- Windows laptop (Chrome/Edge primary browser)
- iPhone (Safari, Substack iOS app, Twitter/X iOS app)
- Single authenticated user (no multi-user support in MVP)

---

## 2. System Architecture

```
Browser Extension / iOS PWA Share Sheet
              ↓
         Ingestion API  (FastAPI)
              ↓
   Content Processing Pipeline
   [fetch → extract → detect → segment → embed]
              ↓
   PostgreSQL + pgvector  (persistent store)
              ↓
   Recommendation Engine  (Python)
              ↓
   Next.js Web App  (reading UI)
```

## 7. Database Schema

Run via Alembic migrations.

```sql
-- Enable pgvector
CREATE EXTENSION IF NOT EXISTS vector;

-- Main content table
CREATE TABLE content (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    source          TEXT,                         -- domain name e.g. "substack.com"
    author          TEXT,
    url             TEXT UNIQUE NOT NULL,
    content_type    TEXT NOT NULL                 -- 'twitter_thread' | 'substack' | 'article' | 'pdf_report' | 'research_paper'
                    CHECK (content_type IN ('twitter_thread','substack','article','pdf_report','research_paper')),
    clean_text      TEXT,
    word_count      INTEGER DEFAULT 0,
    estimated_time  FLOAT DEFAULT 0,              -- minutes
    embedding       vector(1536),                 -- text-embedding-3-small produces 1536-dim
    publish_date    TIMESTAMPTZ,
    status          TEXT DEFAULT 'pending'        -- 'pending' | 'processing' | 'ready' | 'failed'
                    CHECK (status IN ('pending','processing','ready','failed')),
    error_message   TEXT,                         -- populated if status = 'failed'
    created_at      TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX ON content USING hnsw (embedding vector_cosine_ops);
CREATE INDEX ON content (content_type);
CREATE INDEX ON content (status);
CREATE INDEX ON content (created_at DESC);

-- Segments (sub-articles for long content)
CREATE TABLE segments (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    content_id      UUID NOT NULL REFERENCES content(id) ON DELETE CASCADE,
    segment_index   INTEGER NOT NULL,
    text            TEXT NOT NULL,
    word_count      INTEGER NOT NULL,
    estimated_time  FLOAT NOT NULL,               -- minutes
    UNIQUE (content_id, segment_index)
);

CREATE INDEX ON segments (content_id);

-- Reading sessions
CREATE TABLE reading_sessions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    segment_id      UUID NOT NULL REFERENCES segments(id) ON DELETE CASCADE,
    started_at      TIMESTAMPTZ DEFAULT now(),
    ended_at        TIMESTAMPTZ,
    time_spent      FLOAT,                        -- seconds
    completed       BOOLEAN DEFAULT FALSE,
    words_read      INTEGER DEFAULT 0
);

CREATE INDEX ON reading_sessions (segment_id);

-- User stats (single row, upserted)
CREATE TABLE user_stats (
    id              INTEGER PRIMARY KEY DEFAULT 1   -- always 1, single user
                    CHECK (id = 1),
    reading_speed   FLOAT DEFAULT 200,              -- words per minute
    total_words     INTEGER DEFAULT 0,
    total_time      FLOAT DEFAULT 0                 -- seconds
);

INSERT INTO user_stats (id) VALUES (1) ON CONFLICT DO NOTHING;
```

## 9. Content Processing Pipeline

Triggered asynchronously via Celery after ingest. Processing must complete before content appears in recommendations.

```
receive URL
    ↓
detect content type (URL pattern matching)
    ↓
fetch page  →  if PDF: download to R2
    ↓
extract readable content
    │   twitter_thread → scrape DOM (puppeteer or playwright headless)
    │   pdf → pdfplumber or pypdf2
    │   all others → mozilla-readability via Node subprocess OR trafilatura (Python)
    ↓
clean text (strip HTML, normalize whitespace)
    ↓
calculate word_count
    ↓
estimate reading time (word_count / reading_speed)
    ↓
segment article
    ↓
generate embedding (OpenAI text-embedding-3-small)
    │   input: title + first 1500 words
    ↓
store all to database
    ↓
set status = 'ready'
```

**Extraction library priority:**
1. `trafilatura` (Python, handles most articles)
2. `newspaper3k` (fallback for trafilatura failures)
3. Readability via Node subprocess for Substack-specific formatting
4. `pdfplumber` for PDFs

**Failure handling:**
- If extraction yields < 100 words: mark `status = 'failed'`, set `error_message = 'extraction_failed'`
- If URL returns non-200: mark `status = 'failed'`, `error_message = 'fetch_failed'`
- Retry failed items up to 3× with exponential backoff (Celery retry)

## 10. Segmentation Algorithm

```python
DEFAULT_READING_SPEED = 200  # wpm, overridden by user_stats

def segment_content(text: str, reading_speed: int) -> list[Segment]:
    words_per_segment = reading_speed * 6  # 6-minute target segments
    paragraphs = text.split('\n\n')
    segments = []
    current_chunk = []
    current_words = 0

    for para in paragraphs:
        para_words = len(para.split())
        if current_words + para_words > words_per_segment and current_chunk:
            segments.append(build_segment(current_chunk))
            current_chunk = [para]
            current_words = para_words
        else:
            current_chunk.append(para)
            current_words += para_words

    if current_chunk:
        segments.append(build_segment(current_chunk))

    return segments
```

Constraint: Never split mid-sentence. Always break on paragraph boundaries.

## 12. LLM Query Parser

Model: `gpt-4o-mini`

System prompt:
```
You are a query parser for a reading app.
Extract time_budget (minutes), topic, and content_type from the user's message.
content_type must be one of: twitter_thread, substack, article, pdf_report, or null.
Return ONLY valid JSON. No explanation.

Example:
Input: "20 minutes of AI Substack"
Output: {"time_budget": 20, "topic": "AI", "content_type": "substack"}
```

Regex fallback (no LLM):
```python
import re

def regex_parse(query: str) -> dict:
    time_match = re.search(r'(\d+)\s*(min|minute|m\b)', query, re.IGNORECASE)
    time_budget = int(time_match.group(1)) if time_match else None

    content_type = None
    if re.search(r'thread|twitter|tweet', query, re.IGNORECASE):
        content_type = 'twitter_thread'
    elif re.search(r'substack', query, re.IGNORECASE):
        content_type = 'substack'
    elif re.search(r'pdf|paper|report', query, re.IGNORECASE):
        content_type = 'pdf_report'

    return {"time_budget": time_budget, "topic": None, "content_type": content_type}
```

## 15. Twitter/X Bookmark Import

**Approach:** Extension-based DOM scraping (no official API required).

The extension detects when user is on `twitter.com/i/bookmarks` or `x.com/i/bookmarks` and injects a "Import to TimeRead" button into the page DOM.

**Import filter rules (applied before sending):**
```
Import tweet IF any of:
  - is_thread = true  (reply chain from same author, ≥ 2 tweets)
  - contains external article URL
  - tweet length > 280 chars (long-form / premium tweet)

Skip: single short tweets, retweets without comment
```

**Import flow:**
```
user visits bookmarks page
    ↓
extension injects "Import to TimeRead" button
    ↓
user clicks import
    ↓
extension scrapes visible bookmark URLs
    ↓
filter to qualifying tweets
    ↓
show preview: "Found 12 threads and 8 article links. Import?"
    ↓
user confirms
    ↓
batch POST to /ingest (max 20 at a time with 500ms delay)
```

## 17. Offline Support

**Implementation: next-pwa with Workbox**

Cache strategy:
- App shell (HTML/CSS/JS): `CacheFirst`
- API responses for active session: `StaleWhileRevalidate`
- Article text for open session: pre-cached on session start

Pre-cache on session start:
```typescript
// When user clicks "Begin Session", immediately fetch and cache all segment texts
// Store in Cache API under key: session-{id}-segment-{segmentId}
```

Offline behavior:
- If content was pre-cached: read works fully offline
- If not cached: show "This article isn't available offline. Connect to load it."
